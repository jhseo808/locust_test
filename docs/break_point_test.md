# Break Point Test (임계점 테스트)

## 1. 테스트 목적

본 테스트는 동시 사용자 수를 단계적으로 증가시키며  
시스템이 SLA를 안정적으로 유지할 수 있는 **최대 처리 한계(Break Point)** 를  
정량적으로 확인하기 위해 수행되었다.

Spike Test와 달리,  
본 테스트는 급격한 증가가 아닌 **점진적 부하 증가**를 통해  
시스템의 한계를 명확히 드러내는 것을 목표로 한다.

---

## 2. 테스트 대상 및 환경

- 테스트 대상(Host): https://httpbin.org
- 테스트 유형: Break Point Test
- SLA 기준: 응답 시간 3초
- 사용자 증가 패턴:
  - 100 → 200 → 300 → 400 → 500 users
- 각 구간별 충분한 유지 시간 확보

---

## 3. 관찰 지표

Break Point 판단을 위해 다음 지표를 중점적으로 관찰하였다.

- 95% percentile 응답 시간
- Failures 비율
- RPS(Requests Per Second) 변화
- 사용자 수 증가에 따른 지표 변곡점

---

## 4. 결과 해석 기준

- 95% percentile이 SLA 이하로 유지되는 구간  
  → 안정 구간
- 95% percentile이 지속적으로 SLA 초과  
  → 임계점 도달
- Failures 비율이 함께 증가하는 지점  
  → Break Point 확정

---

## 5. 테스트 결과 해석

- 300 users까지는 SLA를 비교적 안정적으로 만족
- 400 users 구간부터 95% percentile이 SLA를 초과하는 스파이크 발생
- Failures 비율이 사용자 증가 구간과 함께 상승

이는 시스템이 완전히 붕괴되지는 않았으나,  
성능 여유가 소진되며 **임계점에 근접**했음을 의미한다.

---

## 6. 결론

Break Point Test를 통해  
시스템이 안정적으로 서비스 가능한 사용자 수와  
성능 저하가 시작되는 임계 구간을 명확히 구분할 수 있었다.

본 결과는 운영 기준 사용자 수 산정 및  
성능 개선 우선순위 결정에 활용 가능하다.
